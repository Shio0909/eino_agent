# RAG and Vector Search

This skill covers the core dense retrieval layer. Keep BM25 fusion in `hybrid-retrieval`, chunk splitting in `chunking-strategies`, GraphRAG in `graphrag-*`, and metrics in `rag-evaluation`.

## Core Retrieval Flow

```text
User query -> query embedding -> vector index search -> metadata filter -> top-k chunks -> grounded answer
```

For this project, the important runtime concepts are:

- Embeddings must be generated by one compatible model family per index.
- Vectors are stored with chunk content and metadata so retrieved chunks can be cited or filtered.
- HNSW accelerates nearest-neighbor search but returns approximate nearest neighbors.
- Retrieval quality depends on embedding model, chunk quality, metadata, top-k, and reranking.

## Embedding Model Selection

| Model | Dims | Best For |
|-------|------|----------|
| `text-embedding-3-large` | 3072 | Highest OpenAI quality; supports dimension reduction |
| `text-embedding-3-small` | 1536 | Cost-effective OpenAI default |
| `voyage-3` | 1024 | Code, legal, finance retrieval quality |
| `gte-Qwen2-7B-instruct` | 3584 | Strong open-source instruction embedding |
| `bge-large-en-v1.5` | 1024 | Strong open-source English baseline |
| `multilingual-e5-large` | 1024 | Multilingual retrieval with query/passage prefixes |

Rules:

- Never mix embeddings from different models in the same vector index.
- Version the embedding model in metadata before changing models.
- Re-embed the whole affected index when model, dimension, or preprocessing changes.
- Benchmark on project queries instead of trusting public leaderboard scores.

## Similarity Metrics

| Metric | Use When | Notes |
|--------|----------|-------|
| Cosine | Default for normalized semantic embeddings | Common with pgvector `vector_cosine_ops` |
| Dot product | Magnitude carries useful signal | Requires model compatibility |
| L2 | Raw geometric distance matters | Less common for text embedding RAG |

In pgvector, cosine distance with `ORDER BY embedding <=> query_vector` ranks lower distance first. A common score conversion is `1 - distance`, but ranking should use the distance operator directly.

## Index Selection by Scale

| Vector Count | Index Type | Notes |
|--------------|------------|-------|
| `<10K` | Flat exact scan | Simple and exact; often fast enough |
| `10K-1M` | HNSW | Strong recall/speed trade-off |
| `1M-100M` | HNSW + quantization or sharding | Watch memory and build time |
| `>100M` | IVF/PQ/DiskANN style systems | Scale trade-offs become dominant |

## HNSW Mental Model

HNSW is a layered small-world graph over vectors:

- Upper layers contain fewer long-range links for fast coarse navigation.
- Lower layers contain denser local links for final nearest-neighbor refinement.
- Search starts from an entry point, greedily moves closer to the query, then explores a candidate set.
- Results are approximate: close vectors are usually reachable, but recall depends on parameters and data distribution.

### HNSW Parameters

| Parameter | Meaning | Trade-off |
|-----------|---------|-----------|
| `m` | Max graph neighbors per vector layer | Higher improves recall, costs memory/build time |
| `ef_construction` | Candidate breadth while building | Higher improves index quality, slows ingestion |
| `ef_search` | Candidate breadth while querying | Higher improves recall, increases latency |

For a concise interview answer: “We store chunk embeddings in pgvector, build an HNSW cosine index, and query with the embedded question ordered by cosine distance. HNSW keeps a navigable graph of nearby vectors, so search can jump through the graph instead of scanning every row.”

## Vector Database Selection

| DB | Strength | Best For |
|----|----------|----------|
| pgvector | Postgres simplicity, transactions, SQL filters | Small/medium systems and teams already on Postgres |
| Qdrant | Filtering, quantization, payload indexing | Production retrieval service with strong metadata filtering |
| Weaviate | Built-in hybrid and schema APIs | Rapid prototyping and multi-modal use cases |
| Elasticsearch/OpenSearch | Mature BM25 and ops ecosystem | Search-heavy systems adding vector retrieval |
| Pinecone | Managed vector infrastructure | Teams avoiding vector DB operations |

## Baseline Retrieval Checks

Before adding hybrid or agentic complexity, verify:

- Top-k contains the expected source chunk for representative questions.
- Metadata filters do not remove relevant chunks.
- Chunk IDs and source metadata are stable enough for citations and updates.
- Retrieval scores separate relevant and irrelevant chunks clearly enough.
- Empty or low-confidence retrieval leads to refusal instead of unsupported generation.
