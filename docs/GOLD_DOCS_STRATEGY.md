# gold_docs 标注策略（公开基准 + 自建评测）

## 结论先说

- **想证明项目对你业务有效**：必须有你自己文档上的 `gold_docs` 标注。
- **想证明方法不是只对你数据有效**：再补公开基准对照（可做加分项）。
- 最稳妥简历表达：**“业务评测集主结论 + 公开基准辅验证”**。

## 1. 公开数据里有没有 `gold_docs`？

有，但形态不完全一样：

- 检索基准（常见）：给 `query -> relevant doc ids`，天然可转成 `gold_docs`
  - BEIR（英文）
  - MIRACL（多语言）
  - C-MTEB 检索任务（中文生态）
- 纯 QA 基准（常见）：多是 `query -> answer`，不一定给文档 ID
  - 这类更适合测“答案正确率”，不直接等于检索 `gold_docs`

所以你的问题答案是：**公开基准有可用的 gold 相关标注，但你自己的系统最终还要落在自建文档上做标注**。

## 2. 为什么你仍然需要“自己标”

因为你面试里最有价值的是：

1. 你这个项目的真实语料是什么（公司知识库、内部文档、网站内容）
2. 在这个真实语料上，哪套方案更好（pipeline/agentic、reranker on/off）
3. 提升是否可复现（固定数据集和评测脚本）

公开基准只能说明“方法一般有效”，不能替代“你的系统在你的语料上有效”。

## 3. 最小可行标注规范（建议）

每条样本建议字段：

- `id`
- `question`
- `category`
- `gold_docs`（核心）
- `answer_keywords`（快速自动评估）
- `expected_answer`（人工复核用）
- `judge_rule`（防止不同标注人标准不一致）

建议先做 60 条：

- 事实问答 25
- 配置/流程 15
- 对比/解释 10
- 无答案问题 10

## 4. 标注成本优化（推荐流程）

1. 先用脚本跑一遍问题集，自动抓取系统返回的 `references` 作为 `candidate_docs`
2. 人工只做“勾选/修正”得到 `gold_docs`
3. 再跑正式评测，产出 Recall/Hit/MRR/nDCG

本仓库已提供辅助命令：

- `go run ./cmd/eval_label -input data/eval_set.jsonl -output data/eval_set_labeled.jsonl -base-url http://localhost:8080 -mode pipeline`

该命令会给每条问题补充 `candidate_docs`，你只需在输出文件里确认并填写 `gold_docs`。

## 5. 什么时候算“测试够用”

- `Retrieval标注覆盖率 >= 80%`
- 样本数 `>= 50`（更建议 80~120）
- 至少 3 组 A/B 对照（同一数据集）
- 每组至少跑 2 次，排除偶然波动

达到上述条件后，写简历里的“提升 X%”会更可信。

## 6. 现在要不要加“垂直领域场景”？

结论建议（按时间预算）：

- 时间紧（<= 2 周）：先保持通用 RAG 项目，但补齐评测闭环（可复现 + 可量化）。
- 时间中等（2~4 周）：加一个“轻量垂直场景”作为加分，不要重构全项目。
- 时间充足（>= 1 个月）：做成“垂直业务 RAG”，含流程化评测与案例演示。

为什么：

- 对暑期实习简历来说，面试官先看的是“你能不能把工程做完整”（功能、评测、对比实验、稳定性）。
- 没有垂直场景并不是致命问题；没有量化证据和可复现实验才是。

## 7. 低成本最优路径（推荐）

如果你觉得手工标注花时间，建议走“80/20”方案：

1. 先做 30 条高质量样本（不是 100 条）
  - 每条都有 `gold_docs`
  - 覆盖 4 类问题：事实、配置流程、对比解释、无答案

2. 用 `eval_label` 先自动给出 `candidate_docs`
  - 人工只做勾选修正，显著减少时间

3. 只跑 2 组关键 A/B
  - `reranker on/off`
  - `rewrite on/off`

4. 输出一页结果表
  - Recall@5、MRR@5、nDCG@5、P95、Error Rate

这样通常 2~4 天就能形成“可写简历 + 可过追问”的结果。

## 8. 垂直场景如何最小化投入

不建议现在大改架构，只建议增加一层“业务包装”：

- 选一个你熟悉且资料易得的方向（如 Go 面试题库、数据库排障、云原生 FAQ）
- 只新增：
  - 一份领域数据集
  - 一份评测集
  - 一页案例演示（典型问题 + 引用来源 + 结果对比）

这会让项目从“通用 RAG Demo”升级为“有业务落点的 RAG 系统”，但不会拖慢主线交付。
